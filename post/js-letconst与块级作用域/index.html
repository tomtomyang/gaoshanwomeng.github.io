1. 作用域对var的处理

先通过一段代码，明确一下作用域机制和js代码的解释执行：

 1function changeA(){
 2   var a = 2;
 3   console.log(a);
 4}
 5
 6var a = 1;
 7changeA();
 8console.log(a);
 9// console output
10// 2
11// 1

模拟解释执行过程：

  1. 全局作用域中，解释器提升var a，将变量a存入全局作用域维护的变量表中，没有赋值；
  2. 全局作用域中，解释器看到函数的声明，将函数changeA存入当前作用域维护的变量表中，并给函数赋值；
  3. 引擎执行a = 1，在全局作用域维护的变量表中查找a的容器，找到了，在容器中存入1；
  4. 引擎执行changeA()，在全局作用域中查找changeA，找到了，执行函数，进入函数作用域；
  5. 函数作用域中，解释器提升var a，将将变量a存入全局作用域维护的变量表中，没有赋值；
  6. 引擎执行a = 2，在函数作用域维护的变量表中查找a的容器，找到了，在容器中存入2；
  7. 引擎执行console.log(a)，在函数作用域维护的变量表中查找a的值，找到了2，交给log()执行；
  8. 函数执行结束；
  9. 引擎执行console.log(a)，在全局作用域维护的变量表中查找a的值，找到了1，交给log()执行；

2. let&const和var的区别

我们可以从无数个资料中查到，let&canst和var具有如下几点不同：

  • let/const存在暂时性死区；
  • let/const不会为全局变量window设置属性；
  • let/const不能重复声明，如果重复声明，在代码编译阶段就会报错；
  • let/var声明的变量可以重新赋值(更改指针的指向)，而const声明的变量不可以重新赋值(不可以更改指针的指向)；

究其原因，是作用域或者说解释器对var和let/const的处理方式不同，导致了let&const与const具有不同的特性。

下面我们就分别来看，是怎样的机制导致了let&const与var的不同，因为let和const只有细微的不同，下面我们只讨论let和var的区别。

1. 暂时性死区

let允许你声明一个作用域被限制在 块级中的变量、语句或者表达式。与 var 关键字不同的是， var声明的变量只能是全局或者整个函数块的。 var 和 let 的不同之处在于后者是在编译时才初始化。

与通过 var 声明的有初始化值 undefined 的变量不同，通过 let 声明的变量直到它们的定义被执行时才初始化。在变量初始化前访问该变量会导致 ReferenceError。该变量处在一个自块顶部到初始化处理的“暂存死区”中。

1function do_something() {
2  console.log(bar); // undefined
3  console.log(foo); // ReferenceError
4  var bar = 1;
5  let foo = 2;
6}

那么我们是不是可以说，解释器进入作用域后，不会对let声明的变量进行提升，其实这种说法不准确，更准确的说法是，let声明的变量的声明被提升了，但是初始化和赋值没有被提升，通俗的说是，已经在内存中开辟了一块空间，但是没有给这块空间起名字，所以你找不到，更没有向这片空间里存储内容。

暂时性死区的本质就是，只要一进入当前作用域，所要使用的变量就已经存在了，但是不可获取，只有等到声明变量的那一行代码出现，才可以获取和使用该变量。

2. 不为window设置属性

我们知道，因为存在变量的遮蔽问题，所以我们在全局作用域使用var声明变量时，会给window对象设置与变量名相同的属性以及值。

而使用let声明的变量不会这样做，那在全局作用域中使用let声明的变量会出现在哪里？

通过上图也可以看到，在全局作用域中，用 let 和 const 声明的全局变量并没有在全局对象中，只是一个块级作用域（Script）中。

3. 不能重复声明

在同一个块中，不能重复声明：

1let x = 1
2let x = 2
3// Uncaught SyntaxError: Identifier 'x' has already been declared

在不同块中，可以重复声明：

1{
2    let x = 1
3    {
4        let x = 2
5    }
6}

3. for循环中的let和var

带着上面的知识，我们来分析一段代码：

1for (var i = 0; i < 5; i++) {
2    setTimeout(() => {
3        console.log(i)
4    }, 1000);
5}
6// 5 5 5 5 5

大家都知道会打印出 5 个 5。如果把 var i 改成 let i，就会分别打印出 0、1、2、3、4：

1for (let i = 0; i < 5; i++) {
2    setTimeout(() => {
3        console.log(i)
4    }, 1000);
5}
6// 0 1 2 3 4

这种区别是为什么？来试着分析一下：

首先，根据网络上的一个说法是for循环可以理解成这样的状态：

1var i;
2for (i = 0; i < 5; i++) {
3    setTimeout(() => {
4        console.log(i)
5    }, 1000);
6}
7// 循环时 i的值从1变到5
8// 点击时 引擎去查找i的引用，在全局作用域下找到，值为5
9// 因此全是5

1let i;
2for (i = 0; i < 5; i++) {
3    setTimeout(() => {
4        console.log(i)
5    }, 1000);
6}
7// 循环时 i的值从1变到5
8// 点击时 引擎去查找i的引用，在全局作用域下找到，值为5
9// 因此全是5

这样分析与结果不符啊，但是按照我们已有的知识，let就是在全局作用域下的，执行点击时，i的值肯定会是5啊！

而事实上，我们运行上面的代码，结果真的是都是5。

既然事实没错，我们对let的理解没错，那就说明事实上的let i; for(i)和for(let i)并不一样，那不一样的点肯定就在let i的位置不在外层作用域，那有没有可能在块级作用域内？

1for (i = 0; i < 5; i++) {
2    let i;
3    setTimeout(() => {
4        console.log(i)
5    }, 1000);
6}
7// undefinrd x 5

看来let i也不是定义在块级作用域内部啊，那它在哪？

于是我去看 MDN 的例子，发现鸡贼的 MDN 巧妙地避开了这个问题，它的例子是这样的：

你看 MDN 的例子，在每次循环的时候用 let j 保留的 i 的值，所以在 i 变化的时候，j 并不会变化。而console.log 的是 j，所以不会出现 5 个 5。

为什么 MDN 要故意声明一个 j 呢，为什么不直接用 i 呢？我猜测 MDN 为了简化知识，隐瞒了什么。

于是我去看了看 ES 文档，其中的 13.7.4.7 章节 清楚地说明了个中缘由，但是由于说得太清楚了，很多人都看不下去，不信你可以试试。

我说一下我的理解：

  1. for( let i = 0; i< 5; i++) 这句话的圆括号之间，有一个隐藏的作用域
  2. for( let i = 0; i< 5; i++) { 循环体 } 在每次执行循环体之前，JS 引擎会把 i 在循环体的上下文中重新声明及初始化一次。
  3. 其他细节就不说了，太细碎了

也就是说，在for设置循环变量的地方存在一个父作用域，循环体{}中是一个子作用域，那么刚才这段代码：

1for (let i = 0; i < 5; i++) {
2    setTimeout(() => {
3        console.log(i)
4    }, 1000);
5}

可以近似的理解为：

1for( let i = 0; i < 5; i++){
2  let i = 隐藏作用域中的i // 看这里看这里看这里
3  setTimeout(() => {
4        console.log(i)
5  }, 1000);
6}

这样才能把一切解释通。

4. let存不存在变量提升

先看第一段代码

1let a = 1;
2{
3    a = 2;
4    console.log(a);
5}
6console.log(a);
7// console output
8// 2
9// 2

1let a = 1
2{
3  	a = 2
4  	let a;
5  	console.log(a);
6}
7console.log(a);
8// console output
9// ReferenceError: Cannot access 'a' before initialization

这个报错是最新版的报错，原本的报错是Uncaught ReferenceError: a is not defined，无论是哪种报错，按照没有提升的思路来分析，外层a的值应该被修改为2，但是没有，而且新版的报错更加清晰的指出了：Cannot access 'a' before initialization，在a被初始化之前，不能获取a，但实际上是存在变量提升的。

  1. MDN 关于 let 是否存在提升的章节，被编辑了两次，第一次说存在提升，第二次说不存在提升（参考 2017 年 3 月 10 号的变更记录）。也就是说 MDN 的维护者都在这个问题上产生过分歧，更何况我们了。
  2. ES 文档里出现了「var/let hoisting」字样。

鉴于此，我认为应该尊重 ES 文档，认为 let 确实存在提升。只不过由于暂时死区的限制，你不能在 let x 之前使用 let。
