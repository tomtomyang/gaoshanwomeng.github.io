1. HTTP 的缺陷

HTTP在通信过程中，存在以下问题：

  • 通信使用明文（不加密），内容可能被窃听
  • 不验证通信方的身份，因此有可能遭遇伪装

这并不安全，HTTPS出现正是为了解决HTTP不安全的问题。

为了保证这些隐私数据能加密传输，让HTTP运行安全的SSL/TLS协议上，即 HTTPS = HTTP + SSL/TLS。SSL 协议位于TCP 协议与各种应用层协议之间，浏览器和服务器在使用 SSL 建立连接时需要选择一组恰当的加密算法来实现安全通信，为数据通讯提供安全支持。

SSL的实现这些功能主要依赖于以下几种技术：

  • 非对称加密：实现身份认证和密钥协商
  • 对称加密：采用协商的密钥对数据加密
  • 摘要算法：验证信息的完整性
  • 数字签名：身份认证和完整性校验

2. 两种加密方式

1. 对称加密

  1. 一个密钥，它可以加密一段信息，也可以对加密后的信息进行解密。
  2. 如果通信双方都各自持有同一个密钥，且没有别人知道，这两方的通信安全当然是可以被保证的（除非密钥被破解）。
  3. 由服务器生成一个密钥并传输给浏览器，那传输过程中密钥被劫持，劫持者就能用密钥解开双方传输的任何内容了。
  4. 如果浏览器内部就预存了网站A的密钥，且可以确保除了浏览器和网站A，这样浏览器要预存好世界上所有HTTPS网站的密钥。

2. 非对称加密

  1. 非对称的密钥分为两个密钥：一把叫做私有密钥，一把叫做公开密钥。
  2. 私有密钥不能让其他任何人知道；公开密钥则可以随意发布，任何人都可以获得。
  3. 发送密文的一方使用公开密钥进行加密处理，对方收到后，用自己的私有密钥进行解密。
  4. 应用到浏览器和服务器的双方通信就是：
    1. 网站服务器拥有公钥A与对应的私钥A；浏览器拥有公钥B与对应的私钥B。
    2. 浏览器把公钥B明文传输给服务器。服务器把公钥A明文给传输浏览器。
    3. 浏览器向服务器传输的内容都用公钥A加密，服务器收到后用私钥A解密。
    4. 服务器向浏览器传输的内容都用公钥B加密，浏览器收到后用私钥B解密。
  5. 非对称加密算法非常耗时。

通过上述操作，可以知道在加密解密过程中，通常是公钥加密，私钥解密，但是并不局限于此：

非对称加密中的公钥和私钥：

第一种用法：公钥加密，私钥解密。用于加解密 第二种用法：私钥签名，公钥验签。用于签名

3. HTTPS 加密方式

非对称加密+对称加密结合

  1. 某网站服务器拥有用于非对称加密的公钥A、私钥A。
  2. 浏览器向网站服务器请求，服务器把公钥A明文给传输浏览器。
  3. 浏览器随机生成一个用于对称加密的密钥X，用公钥A加密后传给服务器。
  4. 服务器拿到后用私钥A解密得到密钥X。
  5. 之后双方所有数据都通过密钥X加密解密即可。

4. 中间人攻击

中间人攻击会在如下场景下发生：

  1. 某网站服务器有用于非对称加密的公钥A、私钥A。
  2. 浏览器向网站服务器请求，服务器把公钥A明文给传输浏览器。
  3. 中间人劫持到公钥A，保存下来，把数据包中的公钥A替换成自己伪造的公钥B（中间人当然也拥有公钥B对应的私钥B）。
  4. 浏览器生成一个用于对称加密的密钥X，用公钥B（浏览器无法得知公钥被替换了）加密后传给服务器。
  5. 中间人劫持后用私钥B解密得到密钥X，再用公钥A加密后传给服务器。
  6. 服务器拿到后用私钥A解密得到对称加密的密钥X。

**中间人掉包了服务器传来的公钥，进而得到了密钥X。**根本原因是 浏览器无法确认收到的公钥是不是网站服务器的。

5. CA 证书

为了解决中间人攻击中 浏览器无法确认收到的公钥是不是网站服务器的 这个问题，CA 证书应运而生。

1. 证书内容

网站在使用HTTPS协议前，需要向CA机构申领一份数字证书，数字证书里含有证书持有者信息、公钥信息等：

前使用最广泛的标准为ITU和ISO联合制定的X.509的 v3版本规范 (RFC5280）, 其中定义了如下证书信息域：

版本号(Version Number）：规范的版本号，目前为版本3，值为0x2；

序列号（Serial Number）：由CA维护的为它所发的每个证书分配的一的列号，用来追踪和撤销证书。只要拥有签发者信息和序列号，就可以唯一标识一个证书，最大不能过20个字节；

签名算法（Signature Algorithm）：数字签名所采用的算法，如：sha256-with-RSA-Encryption；ccdsa-with-SHA2S6；

颁发者（Issuer）：发证书单位的标识信息，如 ” C=CN，ST=Beijing, L=Beijing, O=org.example.com，CN=ca.org。example.com ”；

有效期（Validity）：证书的有效期很，包括起止时间。

主体（Subject） : 证书拥有者的标识信息（Distinguished Name），如：” C=CN，ST=Beijing, L=Beijing, CN=person.org.example.com”；

主体的公钥信息（SubJect Public Key Info）：所保护的公钥相关的信息：

公钥算法 (Public Key Algorithm）公钥采用的算法；

主体公钥（Subject Unique Identifier）：公钥的内容。

颁发者唯一号（Issuer Unique Identifier）：代表颁发者的唯一信息，仅2、3版本支持，可选；

主体唯一号（Subject Unique Identifier）：代表拥有证书实体的唯一信息，仅2，3版本支持，可选：

扩展（Extensions，可选）: 可选的一些扩展。中可能包括：

  • Subject Key Identifier：实体的秘钥标识符，区分实体的多对秘钥；
  • Basic Constraints：一指明是否属于CA;
  • Authority Key Identifier：证书颁发者的公钥标识符；
  • CRL Distribution Points: 撤销文件的颁发地址；
  • Key Usage：证书的用途或功能信息。

此外，证书的颁发者还需要对证书内容利用自己的私钥添加签名， 以防止别人对证书的内容进行篡改。

这里又有一个问题，**证书本身的传输过程中，如何防止被篡改？**答案是使用数字签名。

2. 摘要算法

所谓的摘要就是一段信息或者一个文件通过某个哈希算法（也叫摘要算法）而得到的一串字符（哈希值）。

摘要算法的特点就是不同的文件计算出的摘要是不同的（也有可能相同，但是可能性非常非常低），比如一个1G的视频文件，哪怕只是改动其中一个字节，最后计算得到的摘要也是完全不同的，所以摘要算法通常是用来判断文件是否被篡改过。其还有一个特点就是通过摘要是无法推导出源文件的信息的。

常见的摘要算法有MD5、SHA1、SHA256、SHA512等。

关于摘要算法，有几点需要你明白的：

  • 摘要算法把任意长度的信息，映射成一个定长的字符串；
  • 两个相同的信息，算出的摘要值一定相同；
  • 两个不同的信息，是有可能算出同一个摘要值的，但是几率很小；
  • 摘要算法与加密算法不同，不存在解密的过程，是不可逆的；
  • 摘要算法不用于数据的保密，而是用于数据的完整性校验；

3. 数字签名

了解了摘要算法，还要明确一件事，在非对称加密进行密钥协商之后，客户端还保留了公钥A，服务端还保留了私钥A和公钥A。

数字签名就是利用了非对称加密和摘要算法，来进行信息的完整性校验和身份的校验。

通过上面的图片可以看到，加入数字签名之后，每一次信息中都会包含这样几种数据：

  1. 使用私钥加密的摘要
  2. 原文（使用对称密钥加密）
  3. 使用的哪种摘要算法

这样，服务端每发送一次数据，就要做如下几件事：

  1. 使用摘要算法，将原文生成hash
  2. 使用私钥A加密hash串
  3. 保存使用的摘要算法
  4. 使用对称加密密钥，加密原文

客户端接收到数据，会做如下几件事：

  1. 使用公钥A解密hash串
  2. 使用对称加密密钥解密原文
  3. 使用保存的摘要算法，将原文生成hash
  4. 对比生成的hash和加密出来的hash
