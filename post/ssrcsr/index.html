1. SSR和CSR的概念

SSR(Server Side Rendering)：

SSR是传统的渲染方式，由服务端把渲染好的完整的页面发送给客户端。客户端拿到的是完整的页面，可以直接展示。前后端分离之前很长的一段时间里都是以服务端渲染为主（ASP、JSP、PHP）。

简单理解就是浏览器发送请求后，服务器把客户端网页和数据在后台渲染解析，之后把渲染后的结果返回客户端。

CSR(Client Side Rendering)：

前后端分离的开发模式大大减低了开发难度，提高了开发效率。SPA(single page web application)三大框架Vue、React、Angular被开发人员广泛的接受和使用。因此CSR变成了一种流行的渲染方式。

客户端渲染简单理解就是浏览器发送页面请求，服务器返回的是一个模板页面，这里的服务器不是指我们开发中的后台服务器，而是http服务器，例如Apache、iis等，浏览器从上至下解析过程中需要发送ajax请求获取数据，并把得到的结果添加到页面指定容器中。也就是说，服务器给我们返回的仅仅只是一个 HTML 结构框架，剩下的所有内容都是通过 JS 来产生。

如上代码，服务器端直接给出形如这样的 HTML，浏览器在渲染 body#root 这个节点完成之后，页面的绘制区域其实还是空的，直到 render.js 构建好真实的 DOM 结构之后再 append 到 #root上去。首屏展示出来，必然是发生在 render.js 中网络请求完毕、 js代码执行完成之后，其本质是DOM 的构建者是 JavaScript，而非原生的浏览器。

简而言之，就是数据拼接HTML字符串这件事放在服务端还是客户端造成了两者区别。

2. SSR和CSR各自的优势

2.1 首屏渲染速度

服务器端渲染的首屏渲染速度快，简单来讲就是它不需要来回多次往返于客户端和服务端。

注意：

  1. 这里的首屏加载速度指的是首屏完全加载完成的时间，而不是显示出页面的时间。因为很多情况下，页面的样式即使显示出来了，在表面之下还在进行着很多请求，在这期间页面中的很多功能是无法使用的，也会出现页面样式都出来了，但是鼠标滑轮都没用的情况。而这种情况多数出现在CSR渲染中。

  2. SSR首屏渲染速度快指的是在后台服务器压力不大，能够快速处理请求的情况下，渲染速度会很快。即在理想情况下，理论上的渲染速度会很快。但实际上会受到多种因素的影响，比如说：网速，在线活跃人数，服务器的物理位置等等。

客户端渲染，贴近用户端越近，交互越复杂，它的弊端就越明显，在我们享受 JavaScirpt 给我们带来的无刷新体验和组件化带来的开发效率的同时，由于本质是DOM 的构建者是 JavaScript，而非原生的浏览器，必然会牺牲掉一部分性能。

2.2 SEO

SEO 即 search engine optimization（搜索引擎优化），一般的搜索引擎它的爬虫可能只认 HTML 中的内容，而不能识别 JS 中的内容，因此 CSR 在 SEO 上面就会有巨大的劣势。

对于SSR来说，服务器端渲染的页面在网络中传输的时候，传输的是一个真实的页面。因此，爬虫客户端当爬到我们的页面后，会分析我们给他提供的这个页面，此时，我们页面中的关键数据就会被爬虫给收录了。

对于CSR来说，页面中的关键数据是异步获取，所以在展示完整页面的过程中最少发起两次请求。数据是动态的添加到页面中，不利于爬虫抓取，因此，非常不利于SEO。

然而对于一个网站来说，SEO又至关重要，你的网站别人搜不到，访问量从何谈起。

2.3 开发难度

从上面我们可以知道，SSR是和前后端不分离的开发模式绑定着的，而CSR是和前后端分离的开发模式绑定着的。显而易见的是前后端分离可以大大降低开发难度，这个不再介绍。

2.4 优势对比

CSR优势：

  1. 局部刷新。无需每次都进行完整页面请求
  2. 节约服务器成本。省电省钱，JS 支持 CDN 部署，且部署极其简单，只需要服务器支持静态文件即可
  3. 天生的关注分离设计。服务器来访问数据库提供接口，JS 只关注数据获取和展现

SSR优势：

  1. 首屏渲染速度相对快
  2. SEO
  3. 对于电量不给力的手机或平板，减少在客户端的电量消耗很重要

3. SSR和CSR如何选择

随着技术的进步，CSR模式下SEO问题和首屏加载速度慢问题都可以解决，以vue为例子，百度可以搜索到很多解决的教程，就不再介绍。这里要介绍的是另一项技术。

3.1 前后端同构渲染

3.1.1 理念

Node.js 的出现极大程度的给传统前端赋予了更大的能量，前端的分离也从前期的物理文件的区分转变为职责上的区分，前端开发者从页面仔的噩梦中解脱出来，最重要的是，JavaScript 能在服务器端执行了。在享受这些红利的同时，我们就会不自觉的设想一种方案，它拥有 CSR 的大部分优点，却解决了它的缺点，那就是服务器端输出 HTML，然后由客户端复用该 HTML，继续 CSR 模式，这样岂不是既解决了首屏加载问题和 SEO 问题，又继承了CSR的开发组件化等优点。

如果这样的话，就会有个一致性的问题。我们必须在浏览器端复用服务器端输出的 HTML 才能避免多套代码的适配，刚好传统的模板渲染是可行的，只要选择一套同时支持浏览器和 Node.js 的模板引擎就能搞定。我们写好模板， 在 Node.js 准备好数据，然后将数据灌入模板产出 HTML，输出到浏览器之后由客户端 JavaScript 承载交互，搞定。

按我的理解，同构渲染的基础条件是，使用Node.js作为后端代码或者是作为前后端的中间层，作为SSR中的假后端，用于生成html文件，这样前端拿到的就是完整的html文件，而不是html模板。换一种说法是，原本在CSR模式中，前端的js既需要完成拼接html模板的任务，还需要完成页面功能交互的功能。而加入了Node.js层后，原本的拼接html的任务交给了Node.js来完成，客户端js只需要完成交互任务。

此处的理论为个人理解，如有不当请指正。

3.1.2 实现原理

软件开发中遇到的所有问题，都可以通过增加一层抽象而得以解决。

组件开发的核心概念是 Virtual DOM ，其本质就是为了解决频繁操作 DOM 而在 HTML 之上做的一层抽象，与模板不同的是，它将数据与结构产生交互，最有代表性的要数vue使用的 MVVM 数据流，也就是大家最喜欢vue的双向绑定特征。仿照这种思想，我们将数据与结构用一个抽象的函数来表示： UI = F(data), 其中 UI 为最终产出前端界面，data 为数据，F 则为模板结构或者 Virtual DOM，模板的方式是 F 只执行一遍，而组件方式则为每次 data 改变都会再执行一遍。

无论是模板方式还是组件方式，前后端同构的方案都呼之欲出：我们在 Node.js 端获取数据 ，执行 F 函数，得到 HTML输出给浏览器，浏览器 JavaScript 复用 HTML，继续执行 F 函数，等到数据变化，继续执行 F 函数，交互也得到解决。

按我的理解，这里的F函数，第一次执行在Node.js端，其余次数执行在浏览器js中。也就是说对于F函数，既要保证可以在Node.js中执行，也要保证在vue中执行。再详细一点就是前后端写了相同的一段代码，这就是前后端同构的含义。

3.1.3 通用代码的实现

由于 F 同时需要在浏览器端和服务器端执行，所以对于整个 Vue App，我们需要同时支持两端，也就是通用代码。所以我们需要将 SPA 架构的代码进行改造：

  • 分为两个入口，分为服务端和客户端，只引入通用代码，然后在不同的环境里调用各自的渲染函数。当然，在客户端 ReactDOM.render 会生成 DOM 结构，而服务器端通过 ReactServer.renderToString 将生成 HTML，需要由 HTTP Server 推给前端，各入口处解决特异的环境问题；
  • 通用代码中不可在不判定执行环境的情况下引用 DOM、调用 window、document 这些浏览器特异和引用 global process 这些服务器端特异的操作，这往往是引起 Node.js 服务出问题的根本原因；
  • 为了兼容两端，在选择库时，需要也同时需要支持两端，比如 axios，lodash 等；
  • React 和 Vue 都有生命周期，需要区分哪些生命周期是在浏览器中运行，哪些会在服务器端运行，或者是同时运行，如使用 Redux 或者 Vuex 等库，最好在组件上引入 asyncData 钩子进行数据请求，同时供两端使用；
  • 判定不同的执行环境可以通过注入 process.env.EXEC_ENV 来解决，形如：

3.1.4 构建与运行
  • 在使用 webpack 进行构建时，需要将公共 App 部分打包出来，形成公共代码，由服务器端引入执行，而客户端可以引用打包好的公共代码，再用 webpack 引入之后进行特异处理即可；
  • 需要引入 Node.js 中间层，负责请求数据，提供渲染能力，提供 HTTP 服务，由于 HTML 模板需要在服务端引入，CDN 文件需要自行处理；
  • babel 的使用，可以在浏览器中通用处理，服务端只解决特殊语法，如 jsx、vue template；
